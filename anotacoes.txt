CRIANDO BRANCH
git branch                                → lista as branches, mostrando em qual você está
git checkout -b nomedabranch              → cria E muda pra a branch nova
git checkout main                         → volta pra branch main
git checkout nomedabranch                 → muda para a branch nova
git merge feature/nome-da-feature         → traz as mudanças da branch pra dentro da main
- Convenções comuns de nome: `feature/` (funcionalidade nova), `fix/` (correção), `core/` (manutenção).

OBSERVAÇÃO: para mudar de branch, deve-se salvar os arquivos e fazer o commit antes

- **Page Object Model (POM)**: cada página do site tem sua classe própria, guardando seletores e ações básicas. Se um seletor mudar, corrige em um lugar só.
- **Workflow/Steps**: camada opcional que combina ações de várias páginas em um "passo composto" reutilizável — só vale a pena criar quando uma sequência se repete bastante entre testes.

- `require('../pages/LoginPage')` → "importa" o conteúdo de outro arquivo pra dentro do arquivo atual.
- `module.exports = { LoginPage }` → "exporta" (libera pra fora) o que pode ser usado por quem importar aquele arquivo.
- `../` no caminho significa "sobe uma pasta"; sem `../`, o Node procura em pacotes instalados (`node_modules`).

1. `git status` → mostra o que mudou (vermelho = não adicionado, verde = já adicionado).
2. `git add .` → adiciona todas as mudanças da pasta atual pra baixo.
3. `git commit -m "mensagem"` → salva (tira uma "foto") dessa versão localmente.
4. `git push` → envia essas mudanças pro GitHub.
- `git diff nome-do-arquivo` → mostra linha por linha o que mudou num arquivo específico, antes de dar `add`.
